# Warehouse Routing Challenge

> **Formal Problem Specification**
> Multi-Agent Path Finding on a Grid with Decentralized Memoryless Policies

---

## 1. Overview

This document defines a **deterministic multi-agent warehouse routing problem** on a fixed two-dimensional grid. Participants submit a single memoryless robot policy. The same policy is executed independently by every robot at every simulation tick.

**Objective.** Maximize the number of completed deliveries within a fixed simulation horizon.

Each robot has:

- Its own fixed **base** (home cell).
- Its own current **target item** (a shelf to visit).
- **No persistent memory** across ticks.
- **No communication channel** with other robots.
- Access to the full warehouse map, the positions of all robots, and only its own base and target.

> [!tip] Core Insight
> The challenge is to design a distributed policy that produces efficient *global* warehouse behavior despite each robot only knowing its *own* task.

---

## 2. Grid and Coordinates

The warehouse is represented as a rectangular grid:

$$
G = \{0, 1, \dots, 51\} \times \{0, 1, \dots, 51\}
$$

- The first coordinate is the column $x$, increasing **left to right**.
- The second coordinate is the row $y$, increasing **top to bottom**.
- The top-left cell is $(0, 0)$. The bottom-right cell is $(51, 51)$.
- The walkable warehouse floor is the $50 \times 50$ interior:

$$
W = \{1, 2, \dots, 50\} \times \{1, 2, \dots, 50\}
$$

Each cell has exactly one static type:

```python
class CellType(Enum):
    EMPTY = 0
    SHELF = 1
    BASE  = 2
```

| Cell Type | Passable | Description |
|-----------|----------|-------------|
| `EMPTY`   | Yes      | Walkable corridor |
| `SHELF`   | **No**   | Impassable storage shelf |
| `BASE`    | **No**   | Impassable external base cell (robot home dock) |

---

## 3. Warehouse Layout

The official layout is a fixed $52 \times 52$ grid containing a $50 \times 50$ walkable warehouse floor:

1. **12 vertical double-rows of shelves** in the interior (each shelf block is 2 cells wide and 10 cells tall).
2. **Two-cell-wide vertical corridors** between shelf blocks.
3. **Two-cell-wide horizontal cross-corridors** breaking the shelf blocks every 10 rows.
4. **Two-cell-wide outer corridors** inside the walkable floor on every side.
5. **96 unique base cells** on the external perimeter (24 per side: top, bottom, left, right). These bases are located outside the walkable warehouse floor (row 0, row 51, col 0, col 51).

The base coordinates are fixed by robot id:

| Robot ids | Side | Base coordinate rule |
|-----------|------|----------------------|
| $0 \le r \le 23$ | Top | $b_r = (3 + 2r,\ 0)$ |
| $24 \le r \le 47$ | Bottom | $b_r = (2 + 2(r-24),\ 51)$ |
| $48 \le r \le 71$ | Left | $b_r = (0,\ 2 + 2(r-48))$ |
| $72 \le r \le 95$ | Right | $b_r = (51,\ 3 + 2(r-72))$ |

> [!important] Layout Invariant
> The exact competition layout is generated by the official layout generator and saved as part of the challenge repository. The layout is **public** and **identical** for all teams.

## 4. Robots

Let $N$ be the number of robots:

$$
N = 96
$$

The set of robot identifiers is $R = \{0, 1, \dots, N-1\}$.

Each robot $r \in R$ has a **unique** base cell $b_r \in G$. The mapping $r \mapsto b_r$ is injective:

$$
r_i \neq r_j \implies b_{r_i} \neq b_{r_j}
$$

> [!note]
> Every base cell is impassable. Robots never walk on base cells. Instead, each robot starts on the walkable corridor cell immediately adjacent to its base, and stands there to drop items.

At any tick $t$, robot $r$ has:

| Symbol | Type | Description |
|--------|------|-------------|
| $p_r(t)$ | $(x, y) \in G$ | Current position (always in the walkable corridor) |
| $b_r$ | $(x, y) \in G$ | Fixed base position (located on the perimeter) |
| $q_r(t)$ | $(x, y) \in G$ | Current target item (a shelf cell) |
| $c_r(t)$ | $\{\text{False}, \text{True}\}$ | Carrying flag |

Robots are **point agents** occupying exactly one grid cell. All robots must remain within the walkable $50 \times 50$ warehouse floor ($1 \le x \le 50$ and $1 \le y \le 50$).

---

## 5. Items and Targets

Items are abstract requests associated with shelf cells. The simulator does not track a global inventory of removable objects. Instead, each robot is assigned a **target item position** $q_r(t)$, which corresponds to a shelf cell.

Target positions are selected from:

$$
S = \{g \in G : \operatorname{cell}(g) = \texttt{SHELF}\}
$$

**Key constraints:**

- Each robot knows only its own current target $q_r(t)$.
- It does **not** know the targets of other robots.
- When robot $r$ completes a delivery, the simulator immediately assigns a new target using the official deterministic target generator.
- Multiple robots may simultaneously be assigned targets in the same shelf region or even the same shelf cell.
- Once a robot successfully picks up from shelf cell $q$, that shelf cell is **reserved** until the robot completes the corresponding `DROP`. Other robots assigned to the same shelf cell cannot pick up from it while it is reserved.

---

## 6. Actions

At every tick, each robot must output exactly one action:

```python
class Action(StrEnum):
    WAIT   = "wait"
    UP     = "up"
    DOWN   = "down"
    LEFT   = "left"
    RIGHT  = "right"
    PICKUP = "pickup"
    DROP   = "drop"
```

Movement displacement vectors:

| Action | $\Delta x$ | $\Delta y$ |
|--------|-----------|-----------|
| `UP`    | $0$       | $-1$      |
| `DOWN`  | $0$       | $+1$      |
| `LEFT`  | $-1$      | $0$       |
| `RIGHT` | $+1$      | $0$       |

The actions `WAIT`, `PICKUP`, and `DROP` produce zero displacement.

---

## 7. Action Validity

### 7.1 Movement Validity

A movement action proposed by robot $r$ at tick $t$ defines an **intended destination**:

$$
d_r(t) = p_r(t) + \Delta(a_r(t))
$$

A movement action is **statically invalid** if any of the following hold:

1. $d_r(t) \notin G$ (out of bounds).
2. $d_r(t) \notin W$ (outside the walkable warehouse floor).
3. $\operatorname{cell}(d_r(t)) = \texttt{SHELF}$ (target cell is a shelf).
4. $\operatorname{cell}(d_r(t)) = \texttt{BASE}$ (target cell is an external base).

If a movement action is statically invalid, the robot's intended destination is replaced by its current position:

$$
d_r(t) := p_r(t)
$$

Equivalently, the invalid movement becomes `WAIT`.

### 7.2 Pickup Validity

`PICKUP` succeeds for robot $r$ at tick $t$ if and only if **all** of the following conditions hold:

1. $c_r(t) = \text{False}$ (not already carrying an item).
2. The robot is **orthogonally adjacent** to its own target item:

$$
\|p_r(t) - q_r(t)\|_1 = 1
$$

3. $q_r(t) \in S$ (target is a valid shelf cell).
4. No other robot is already carrying an item picked up from $q_r(t)$.
5. If multiple robots submit otherwise valid `PICKUP` actions for the same shelf cell in the same tick, only the lowest `robot_id` succeeds; the others are blocked with reason `shelf_locked`.

**On success:** $c_r(t+1) = \text{True}$.
**On failure:** $c_r(t+1) = c_r(t)$ (no change).

`PICKUP` never changes position.

While carrying, `target_item_position` remains the shelf cell where the item was picked up. The robot should use `carrying_item=True` and its own `base_position` to decide that its navigation goal is now the base-adjacent drop cell.

### 7.3 Drop Validity

`DROP` succeeds for robot $r$ at tick $t$ if and only if:

1. $c_r(t) = \text{True}$ (robot is carrying an item).
2. The robot is **orthogonally adjacent** to its own base cell:

$$
\|p_r(t) - b_r\|_1 = 1
$$

**On success:**
1. The team score increases by $+1$.
2. $c_r(t+1) = \text{False}$.
3. The simulator assigns robot $r$ a new target item.

**On failure:** $c_r(t+1) = c_r(t)$ (no change).

`DROP` never changes position.

## 8. Collision Rules

Collision resolution is **deterministic** and **simultaneous**.

First, all submitted actions are converted into intended destinations $d_r(t)$ using the static movement validity rules above. A robot that submits `WAIT`, `PICKUP`, `DROP`, or an invalid movement is **stationary**: $d_r(t) = p_r(t)$. A robot is **moving** iff $d_r(t) \neq p_r(t)$. Only moving robots can be blocked by a dynamic conflict; a stationary robot always keeps its cell (and its `PICKUP`/`DROP` is never cancelled by another robot bumping into it).

### 8.1 Edge Conflict

An **edge conflict** occurs when two robots intend to swap positions:

$$
d_{r_i}(t) = p_{r_j}(t) \quad \text{and} \quad d_{r_j}(t) = p_{r_i}(t), \quad r_i \neq r_j
$$

**Resolution:** Both robots are **blocked** and remain at their current positions for this tick. Edge conflicts are detected on the originally intended destinations.

### 8.2 Vertex Conflict

A **vertex conflict** occurs when a moving robot's destination coincides with the final cell of any other robot. Because blocking a robot returns it to its start cell — which may in turn collide with a follower — the rule is applied **iteratively to a fixpoint**:

> Repeatedly select any moving, not-yet-blocked robot $r$ whose destination $d_r(t)$ is shared by the final cell of another robot, and block it (set its final cell back to $p_r(t)$). Repeat until no such robot remains.

When two or more moving robots contend for the **same** cell, **all** of them are blocked (there is no winner). This process is monotone, terminates in at most $N$ rounds, and its result does not depend on the order in which robots are examined.

> [!example] Following Chain
> Robots A, B, C occupy $(1,1), (2,1), (3,1)$. A and B both move RIGHT; C waits. Because C stays, B cannot enter $(3,1)$, so B is blocked back to $(2,1)$; that in turn blocks A back to $(1,1)$. **Result:** nobody moves. If instead C also moves RIGHT into the empty cell $(4,1)$, the whole chain advances by one.

### 8.3 Blocking Result

$$
p_r(t+1) =
\begin{cases}
p_r(t), & \text{if robot } r \text{ is blocked by any conflict} \\
d_r(t), & \text{otherwise}
\end{cases}
$$

This resolution guarantees the core invariant: **no two robots ever occupy the same cell**, and **no two robots ever swap cells**, after any tick.

> [!example] Edge Conflict Illustration
> Robot A at cell $(3, 5)$ moves RIGHT. Robot B at cell $(4, 5)$ moves LEFT. Both intend to swap. **Result:** both stay in place.

## 9. Tick Resolution Order

Each tick $t$ is resolved in this exact sequential order:

| Step | Operation |
|------|-----------|
| 1 | Construct one observation per robot |
| 2 | Call the submitted policy once per robot |
| 3 | Convert invalid movement actions to `WAIT` |
| 4 | Resolve edge conflicts, then vertex conflicts to a fixpoint (§8) |
| 5 | Apply non-conflicting movement actions |
| 6 | Resolve `PICKUP` actions, including shelf-reservation conflicts |
| 7 | Resolve `DROP` actions |
| 8 | Assign new targets to robots that completed deliveries |
| 9 | Advance to tick $t+1$ |

> [!warning] Timing Constraints
> - If a robot submits `PICKUP` or `DROP`, it does **not** move during that tick.
> - A robot that moves adjacent to its target at tick $t$ may not pick up until tick $t+1$.
> - A robot that moves adjacent to its base at tick $t$ may not drop until tick $t+1$.

## 10. Observation Model

The policy is **memoryless** and **decentralized**. The same pure function is called independently for each robot.

At tick $t$, robot $r$ receives:

```python
@dataclass(frozen=True)
class Observation:
    tick: int
    robot_id: int
    position: tuple[int, int]
    base_position: tuple[int, int]
    target_item_position: tuple[int, int]
    carrying_item: bool
    grid: tuple[tuple[CellType, ...], ...]
    all_robot_positions: dict[int, tuple[int, int]]
```

**Included in observation:**

| Field | Description |
|-------|-------------|
| `tick` | Current simulation tick |
| `robot_id` | This robot's unique identifier |
| `position` | This robot's current $(x, y)$ |
| `base_position` | This robot's home base $(x, y)$ |
| `target_item_position` | This robot's current target shelf $(x, y)$ |
| `carrying_item` | Whether this robot is carrying an item |
| `grid` | Complete static warehouse map |
| `all_robot_positions` | Positions of **all** robots |

**Excluded from observation:**

- Other robots' target item positions.
- Other robots' carrying flags.
- Other robots' base positions.
- Other robots' previous positions or intended actions.
- Any global shared memory.
- Any private memory from previous calls.

> [!abstract] Information Asymmetry
> Each robot can see *where* all other robots are, but not *what they are doing* or *where they are going*. Other robots are visible moving obstacles with hidden intentions.

---

## 11. Memoryless Policy Requirement

A submitted policy must be a **pure function** of the current observation:

$$
\pi : \mathcal{O} \rightarrow \mathcal{A}
$$

The policy must not depend on any persistent internal state across ticks.

**Official interface:**

```python
def act(observation: Observation) -> Action:
    """Return exactly one action for the observing robot."""
```

**Forbidden behaviors:**

- Module-level mutable state used as memory.
- Class instances with persistent fields.
- Reading or writing files during simulation.
- Network access.
- Randomness not derived exclusively from the current observation.
- Inspecting hidden simulator state.

Deterministic pseudo-random behavior is allowed only if it is a pure function of the observation:

```python
def deterministic_coin_flip(observation: Observation) -> bool:
    value = hash((observation.tick, observation.robot_id, observation.position))
    return value % 2 == 0
```

## 12. Submission Interface

Each team submits a single Python file containing an `act` function:

```python
from warehouse_api import Action, Observation

def act(observation: Observation) -> Action:
    bx, by = observation.base_position
    if bx == 0:
        base_drop_pos = (1, by)
    elif bx == 51:
        base_drop_pos = (50, by)
    elif by == 0:
        base_drop_pos = (bx, 1)
    else:
        base_drop_pos = (bx, 50)

    if not observation.carrying_item:
        goal = observation.target_item_position
    else:
        goal = base_drop_pos

    x, y = observation.position
    gx, gy = goal

    if not observation.carrying_item and abs(x - gx) + abs(y - gy) == 1:
        return Action.PICKUP

    if observation.carrying_item and observation.position == base_drop_pos:
        return Action.DROP

    if gx > x:
        return Action.RIGHT
    if gx < x:
        return Action.LEFT
    if gy > y:
        return Action.DOWN
    if gy < y:
        return Action.UP
    return Action.WAIT
```

> [!caution]
> The baseline above is intentionally incomplete. It ignores shelves, other robots, collisions, and congestion.

## 13. Simulation Horizon

The official evaluation runs for a fixed number of ticks:

$$
T = 10{,}000
$$

The simulator starts at tick $0$ and ends after resolving tick $T - 1$. The score is computed after the final tick.

---

## 14. Scoring

Let $D_r$ be the number of successful deliveries completed by robot $r$ during the simulation.

**Primary score:**

$$
\text{Score} = \sum_{r \in R} D_r
$$

Higher is better.

**Tie-breaking** (lexicographic order):

| Priority | Criterion | Direction |
|----------|-----------|-----------|
| 1 | Completed deliveries | Higher is better |
| 2 | Blocked movement attempts | Lower is better |
| 3 | Sum of remaining Manhattan distances | Lower is better |
| 4 | Submission timestamp | Earlier is better |

The remaining distance for robot $r$ at the end of the simulation:

$$
\rho_r =
\begin{cases}
\displaystyle\min_{u \in \mathcal{A}(q_r)} \|p_r(T) - u\|_1, & \text{if } c_r(T) = \text{False} \\[6pt]
\|p_r(T) - b_r\|_1, & \text{if } c_r(T) = \text{True}
\end{cases}
$$

where $\mathcal{A}(q_r)$ is the set of orthogonally adjacent passable cells around target shelf $q_r$.

---

## 15. Target Generation

The official target generator is **deterministic**.

- At the start of a simulation, each robot is assigned one target shelf position.
- When a robot successfully drops an item at its own base, it is immediately assigned a new target.

The target sequence for robot $r$ is generated from:

$$
\text{seed}_r = H(\text{global\_seed},\; r)
$$

where $H$ is the official deterministic hash function.

> [!note]
> The full list of generated target positions is **not** provided to participants during evaluation. A robot only observes its current target.

---

## 16. Determinism and Reproducibility

For a fixed:

- Warehouse layout
- Initial robot positions (each robot starts at its own base)
- Global seed
- Team submission
- Python version and package environment

the simulator **must** produce exactly the same replay and score.

Any source of nondeterminism in a submission is invalid unless it is derived purely from the current observation.

## 17. Evaluation Rounds

The challenge is run in repeated official **evaluation rounds** (e.g., every 30 minutes):

| Step | Action |
|------|--------|
| 1 | Each team submits one `act` implementation |
| 2 | The official harness evaluates each submission |
| 3 | Scores and replays are published |
| 4 | The **source code** of all accepted submissions is shared with all teams |
| 5 | Teams may fork, combine, and improve any published submission |

> [!important] Forced Communication
> After each evaluation round, every team's code becomes public. Teams are encouraged to fork the best-performing submissions, combine strategies, and iterate. This mechanism transforms the competition into a collective optimization process.

---

## 18. Illegal Behavior

A submission is **invalid** if it:

- Attempts to access the filesystem during simulation.
- Attempts network access.
- Uses wall-clock time to affect decisions.
- Uses process-global mutable state as memory.
- Mutates the observation object.
- Imports hidden simulator internals.
- Spawns subprocesses.
- Uses non-standard dependencies not included in the official environment.
- Exceeds the per-action time limit.

Invalid submissions receive score $0$ for the round.

---

## 19. Computational Limits

The policy is called once per robot per tick. For $N = 96$ and $T = 10{,}000$:

$$
N \cdot T = 960{,}000 \text{ policy calls per evaluation}
$$

Each call must be fast.

**Official harness enforces:**

| Constraint | Description |
|------------|-------------|
| Total simulation time | Maximum wall-clock time for the full run |
| Per-tick time | Maximum wall-clock time for all 96 policy calls per tick |
| Compute | CPU-only, no GPU access |
| Environment | Fixed Python version, pinned dependencies |

Exact limits will be published before the event.

## 20. Design Notes

The problem intentionally disallows central planning and persistent memory.

Even though each robot sees all current robot positions, it knows only its own base, target, and carrying state. Other robots are visible moving obstacles with **hidden intentions**.

This creates a distributed coordination problem where strong solutions may leverage:

- **Deterministic traffic conventions** (e.g., keep-right rules in corridors).
- **Priority protocols** based on robot ID, position, carrying state, or tick parity.
- **Local congestion avoidance** (yielding when corridors are dense).
- **Path planning** treating other robots as dynamic obstacles.
- **Deadlock resolution** using deterministic tie-breaking.
- **Symmetry-breaking** from robot ID and tick number.

> [!abstract] Summary
> The challenge is simple to state but difficult to master: each robot must behave locally in a way that produces efficient global flow.
